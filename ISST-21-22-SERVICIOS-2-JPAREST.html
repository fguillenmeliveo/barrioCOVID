Persistencia con Spring Boot y JPA

Objetivos

El Trabajo Fin de Grado (TFG) es el modelo más importante de la aplicación, ya que el objetivo de la aplicación es manejar el ciclo de vida de estos. Cada proyecto tiene un único autor (un estudiante), que sólo puede tener un proyecto, por lo que podemos usar el modelo TFG para almacenar la información tanto del proyecto como del estudiante (es una simplificación del modelo relacional que resulta muy conveniente).

Diseñaremos la capa de persistencia de la aplicación Web, para ello definiremos el modelo de datos de la aplicación, lo convertiremos en entidades JPA y definiremos el objeto de acceso a los datos (Data Access Object - DAO) para leer y escribir objetos de una base de datos relacional (H2) usando el soporte que proporciona Spring Boot.

La implementación moderna de las arquitecturas de servicio requiere implementar servicios sencillos, accesibles mediante interfaces regulares y protocolos estandarizados, con formatos de transferencia de datos ligeros. Tal como lo define DDD, manejaremos nuestras entidades de persistencia a los recursos como servicios REST a los que se ofrece el acceso mediante HTTP y JSON para la transferencia de datos. JAX-RS es el API estándar para todo ello. La siguiente figura muestra el esquema del servicio REST que contiene todas las operaciones de persistencia de TFGs.

Spring Boot como soporte a la capa de persistencia

Usaremos H2 como servicio de gestión de bases de datos relacionales. Ahora necesitamos una biblioteca para conectar nuestra aplicación a H2; esta biblioteca debe proporcionar clases y métodos a los que podamos llamar desde nuestro código, convirtiéndose en el “proveedor de servicio de persistencia”. El API de Persistencia de Java (JPA) define las clases e interfaces que estas bibliotecas deben ofrecer, de manera que pudiéramos usar cualquier proveedor del mercado sin necesidad de cambiar el código de nuestra aplicación. Usaremos Hibernate como proveedor, así que es esta biblioteca la que hará las interacciones con la fuente de datos (H2). Toda la información que sea necesario aportar para configurar la base de datos se puede escribir en el fichero src/main/resources/application.properties

#H2 settings

spring.h2.console.enabled=true

spring.h2.console.path=/h2-console

spring.datasource.url=jdbc:h2:mem:isst

spring.datasource.driverClassName=org.h2.Driver

spring.datasource.username=sa

spring.datasource.password=

spring.jpa.database-platform=org.hibernate.dialect.H2Dialect

spring.jpa.hibernate.ddl-auto=create

server.port=8080

# for running without custom REST implementation

spring.data.rest.base-path=/api

Modelo de datos

En nuestro proyecto Java, las clases se organizan en paquetes/carpetas (packages) dentro de la carpeta src/main/java. Para crear uno nuevo, puede marcar el proyecto y seleccionar “New>Package”. Las clases que actúan como modelos deben:

  • estar en el paquete es.upm.dit.isst.tfgapi.model, 
  • contener un constructor público sin parámetros,
  • y tener métodos accesores y modificadores para todos los atributos.

Puede programar automáticamente los accesores, modificadores, constructor con y sin parámetros, equals, toString y añadir comentarios de documentación de cada clase, una vez definidos sus atributos, usando los atajos de VSC (menú contextual Source Actions). A medida que vaya programando verá que aparecen consejos o marcas de error que puede resolver mediante “code actions” (cmd-. en mac os).

Una vez definidas las clases, las convertiremos en entidades respaldadas en la base de datos con la ayuda de etiquetas JPA (Java Persistence API). Estas anotaciones permiten representar objetos Java en forma de registros de una base de datos (normalmente mediante una tabla por clase). Las anotaciones se ponen justo delante del elemento que deseamos anotar; las que usaremos en el proyecto son (importadas todas de javax.persistence):

  1. @Entity: sobre la declaración de la clase, declara que esta clase será una entidad para representar en la base de datos.
  2. @Id: se coloca sobre el atributo que actuará como clave primaria del modelo (el atributo debe ser una referencia a objeto-no funciona con tipos primitivos).
  3. @Lob: (Large object binary) se coloca sobre atributos que representen objetos binarios, tales como fotos, documentos, etc.

La clase TFG representa a un estudiante y su proyecto fin de grado. Los siguientes atributos son necesarios:

  1. email: email del alumno e identificador único, con la anotación @Id.
  2. pass: contraseña del alumno.
  3. nombre: nombre completo del alumno.
  4. titulo: título del trabajo.
  5. status: estado en el que se encuentra el trabajo, representado con un entero.
  6. memoria: en esta variable se almacenará el archivo con la memoria del trabajo, debe ser de tipo byte[] (array de bytes) y utilizar la anotación @Lob.
  7. nota: nota final del trabajo.
  8. tutor: tutor del proyecto. Es una variable String y contiene el email del tutor.

Data Access Object - DAO

Para realizar las operaciones de persistencia asociadas al modelo (escribir o leer de la base de datos) utilizaremos unas clases conocidas como DAO (Data Access Object). Spring Boot nos ayudará en esta labor, porque contiene una interfaz genérica que proporciona las operaciones más habituales:

public interface CrudRepository<T,ID> extends Repository<T,ID> {

long        count();

void        delete(T entity);

void        deleteAll();

void        deleteById(ID id);

boolean        existsById(ID id);

Iterable<T>        findAll();

Optional<T>        findById(ID id);

<S extends T> S        save(S entity);

}

Corresponde al conocido patrón CRUD (create-read-update-delete) de operaciones de interacción con la base de datos. También usa el patrón fachada que separa la definición de las operaciones -la interfaz- de su implementación mediante una clase completa. En nuestro caso bastará con definir nuestra interfaz que extiende a la anterior con los tipos TFG y String (el del campo clave de nuestra entidad). Estará en el paquete es.upm.dit.isst.tfgapi.repository y necesitará importar su clase TFG, java.util.List y org.springframework.data.repository.CrudRepository.

public interface TFGRepository extends CrudRepository<TFG, String> {

    List<TFG> findByTutor(String tutor);

}

De esta forma tenemos una interfaz TFGRepository con todas las operaciones más la de obtener la lista de TFGs que tienen a un profesor como tutor. Pero aún más: SB es capaz de aportar una clase completa que implementa a la interfaz con los cuerpos de todos los métodos, incluido findByTutor (si se sigue un convenio para nombrar a los métodos que se definan de forma adicional).

Por el mero hecho de haber incluido en pom.xml las bibliotecas de SB para acceso REST, y definido TFGRepository, tendremos un servicio funcional en cuanto haya una clase con método main y anotada como aplicación SB.

@SpringBootApplication

public class SpringBootReactApplication {

}

Si arrancamos el servidor, podremos acceder a las direcciones http://localhost:8080/api y http://localhost:8080/api/tFGs si se ha definido en application.properties la variable spring.data.rest.base-path=/api. Igualmente la base de datos h2 (accesible en http://localhost:8080/h2-console) está corriendo con un esquema según la entidad definida.

./mvnw clean install -DskipTests=true spring-boot:run

En realidad, nos interesa poder controlar mejor las operaciones que vamos a ofrecer como primitivas del servicio REST de persistencia. Concretamente:

Primitiva HTTP y ruta

Semántica

@PostMapping("/tfgs")

@GetMapping("/tfgs/{id}")

El código correspondiente a la implementación es muy sencillo, aunque hace uso de elementos de Java que aparecieron a partir de la versión 8, como el método map que toma como parámetro una expresión lambda (que define una función), y Optional<T> que es una clase que recubre a una referencia que puede ser nula, y sobre la que se aplica map.

@RestController

public class TFGController {

    private final TFGRepository tfgRepository;

    public static final Logger log = LoggerFactory.getLogger(TFGController.class);

    public TFGController(TFGRepository t) {

        this.tfgRepository = t;

    }

    @GetMapping("/tfgs")

    List<TFG> readAll() {

      return (List<TFG>) tfgRepository.findAll();

    }

 

    @PostMapping("/tfgs")

    ResponseEntity<TFG> create(@RequestBody TFG newTFG) throws URISyntaxException {

      TFG result = tfgRepository.save(newTFG);

      return ResponseEntity.created(new URI("/tfgs/" + result.getEmail())).body(result);

    }

 

    @GetMapping("/tfgs/{id}")

    ResponseEntity<TFG> read(@PathVariable String id) {

      return tfgRepository.findById(id).map(tfg ->

         ResponseEntity.ok().body(tfg)

      ).orElse(new ResponseEntity<TFG>(HttpStatus.NOT_FOUND));

    }

    @PutMapping("/tfgs/{id}")

    ResponseEntity<TFG> update(@RequestBody TFG newTFG, @PathVariable String id) {

      return tfgRepository.findById(id).map(tfg -> {

        tfg.setNombre(newTFG.getNombre());

        tfg.setTitulo(newTFG.getTitulo());

        tfg.setTutor(newTFG.getTutor());

        tfg.setStatus(newTFG.getStatus());

        tfg.setNota(newTFG.getNota());

        tfg.setMemoria(newTFG.getMemoria());

        tfgRepository.save(tfg);

        return ResponseEntity.ok().body(tfg);

      }).orElse(new ResponseEntity<TFG>(HttpStatus.NOT_FOUND));

    }

    @DeleteMapping("tfgs/{id}")

    ResponseEntity<TFG> delete(@PathVariable String id) {

      tfgRepository.deleteById(id);

      return ResponseEntity.ok().body(null);

    }

    @GetMapping("/tfgs/profesor/{id}")

    List<TFG> readTutor(@PathVariable String id) {

      return (List<TFG>) tfgRepository.findByTutor(id);

    }

    @PostMapping("/tfgs/{id}/incrementa")

    ResponseEntity<TFG> incrementa(@PathVariable String id) {

      return tfgRepository.findById(id).map(tfg -> {

        tfg.setStatus(tfg.getStatus() + 1);

        tfgRepository.save(tfg);

        return ResponseEntity.ok().body(tfg);

      }).orElse(new ResponseEntity<TFG>(HttpStatus.NOT_FOUND));  

    }

}

Observe el uso de las anotaciones @GetMapping, @PostMapping para asociar los métodos de servicio a las rutas y métodos HTTP, y también @PathVariable para convertir una parte de la ruta en un parámetro del método, y @RequestBody para leer una entidad del cuerpo de la petición HTTP.

Se hace uso de las anotaciones de SB, como:

  • @RestController para identificar la clase como una implementación de servicios REST
  • @GetMapping, @PostMapping, @PutMapping, @DeleteMapping se colocan antes de los métodos para asignar la ruta y método HTTP que activa a cada uno,
  • @PathVariable toma una parte de la ruta y la inserta como un parámetro del método,
  • @RequestBody indica que el parámetro del método debe tomarse del cuerpo del mensaje de petición HTTP.

Será necesario importar clases de los paquetes: org.slf4j, java.net, java.util, org.springframework.http, org.springframework.web.bind.annotation.

SB analiza el código de la clase y añade el código extra necesario para realizar conversiones de/a mensajes HTTP, objetos JSON, objetos Java y valores de retorno. Para la importación de clases en VCS pinche en el nombre y haga cmd. El atributo al que se da valor en el constructor es un repositorio de TFGs; en este punto SB realiza la inyección de beans, así que crea un objeto implementación de TFGRepository y lo enlaza aquí.

Puesto que las rutas son diferentes de la definida en application.properties para la interfaz REST básica, ahora contamos con dos puntos de acceso al servicio de persistencia de repositorios.

Pruebas de la capa de persistencia

Se puede lanzar el servicio de esta forma:

./mvnw clean install spring-boot:run -DskipTests=true

Y después de lanzarlo hay varias opciones para comprobar que funciona correctamente:

  • http://localhost:8080/h2-console abre una consola para interactuar directamente con el servidor h2 en memoria que se lanza con el servicio, desde un navegador Web
  • http://localhost:8080/tfgs lanza una operación GET a esa ruta para obtener todos los TFGs del sistema, o un solo poniendo su clave a continuación, desde un navegador Web (atención porque no funcionará si tiene la app cliente que se menciona en la sección siguiente)
  • pruebe a incluir esta dependencia en su fichero pom.xml y regenerar el proyecto

       <dependency>

            <groupId>org.springdoc</groupId>

            <artifactId>springdoc-openapi-ui</artifactId>

            <version>1.6.14</version>

        </dependency>

esta biblioteca genera documentación sobre su API-REST automáticamente, en la dirección http://localhost:8080/swagger-ui.html

  • puede ejecutar órdenes con curl (un cliente Web de texto por línea de órdenes)
  • curl localhost:8080/api
  • curl localhost:8080/tfgs
  • curl -H "Content-Type: application/json" -d '{ "email": "email1@alumnos.upm.es", "pass": "asdf", "nombre": "nombre", "titulo": "asfsgfsfsdd", "tutor": "juancarlos.duenas@upm.es", "status": 1, "memoria": null, "nota": 0.0 }' -X POST http://localhost:8080/tfgs
  • curl localhost:8080/tfgs
  • curl -H "Content-Type: application/json" -X DELETE http://localhost:8080/tfgs/email1@alumnos.upm.es
  • curl localhost:8080/tfgs
